## 一、设计哲学：三源融合

### 我们从哪里来

| 来源 | 吸收了什么 | 没有吸收什么 | 为什么不要 |
|------|-----------|-------------|-----------|

| **GSD** | (待吸收) 独立上下文策略、上下文水位监控 | 44 命令膨胀、6 平台广度、自主模式 | 克制是美德；质量优先于覆盖面 |
| **Autoresearch** | (待吸收) RL 迭代循环、固定评估环境、keep/discard 纪律、实验记忆、双层优化 | 无人值守全自动 | 软件开发比 LLM 训练需要更多人类判断 |

### v1.1 的核心理念

**"人编程程序，程序编程代码"** — 借用 Karpathy 的表述：

```
外层循环 (人类驱动)：优化 CLAUDE.md / Skills / 评估标准
  └── 内层循环 (Agent 驱动)：修改代码 → 运行评估 → keep/discard → 记录实验 → 重复
```

这是 AlataFlow 与三个参考项目最本质的区别：
- GSD 解决"怎么高效执行"（吞吐量）
- Superpowers 解决"怎么不偷懒"（纪律）
- Autoresearch 解决"怎么持续变好"（迭代）
- **AlataFlow 解决"怎么不遗忘 + 持续进化"（记忆 + 遗传）**

v1.1 的定位：**在记忆和进化的骨架上，嫁接 RL 迭代循环**。

## 三、核心新增：RL 实验循环

### 3.1 设计灵感

Autoresearch 的天才之处：

```python
# Autoresearch 的内层循环（伪代码）
while True:
    idea = agent.think(history=results_tsv, code=train_py)
    git_commit(idea)
    result = run_experiment(timeout=300s)    # 固定时间预算
    log(results_tsv, result)                 # 实验记忆
    if result.metric < best.metric:          # 严格改进
        keep()                               # 推进分支
    else:
        discard()                            # git reset
```

### 3.2 映射到软件开发

| Autoresearch 概念 | AlataFlow 映射 |
|-------------------|---------------|
| `train.py` (可修改的代码) | 当前 Task Space 的源代码 |
| `prepare.py` (不可修改的评估) | `task_plan.md` 中的 Verification Commands |
| `val_bpb` (单一标量指标) | 验证通过率 (pass/total) + 可选自定义指标 |
| `results.tsv` (实验日志) | `.plans/<slug>/experiments.tsv`（任务本地） |
| `program.md` (Agent 策略) | CLAUDE.md + Skills (外层循环由人类优化) |
| `git commit + reset` | 临时实验分支 commit + cherry-pick (keep) / reset --hard + clean (discard) |
| 5 分钟时间预算 | 每轮实验默认 5 分钟超时（`--timeout 300`，可配置） |
| "NEVER STOP" | `/alata:experiment --rounds N` (可控的自主迭代) |

### 3.3 实验循环技能：`/alata:experiment`

**新增 Skill**：适用于优化型任务（性能调优、重构、Bug 修复迭代）

```
/alata:experiment --rounds 5 --metric "npm test 2>&1 | grep 'duration' | awk '{print \$NF}'" --metric-direction lower
```

**Git 隔离模型**：

实验在 **当前 Task Space 的 worktree 内** 运行，在一次性分支 `experiment/<slug>-<timestamp>` 上操作。
这是对 Task Space 不变量的 **受控扩展**：原始不变量为"一个 Task Space = 一个 git worktree = 一个稳定分支"，实验模式在 worktree 内临时切换分支（`git checkout -b experiment/...`），但保证循环结束后切回原工作分支并删除临时分支，恢复不变量。实验期间 worktree 目录不变，聚焦上下文不变——变的只是 HEAD 指向。
实验分支从工作分支 fork 出来，keep 的变更通过 cherry-pick 回到工作分支；discard 的变更通过 `git reset --hard + git clean -fd` 彻底丢弃（包括未跟踪文件和构建产物）。

```
Task Space worktree (.worktrees/<slug>/):
  工作分支 (alataflow/feat/<slug>):   ──A──B──────────────────C (cherry-pick)──D (cherry-pick)──
                                             \               /                /
  experiment/<slug>-<ts> (临时分支):          └──X (discard)──Y (keep)────────Z (keep)── (删除)
```

**discard 操作**：`git reset --hard HEAD~1 && git clean -fd`（reset 回退 commit，clean 清除未跟踪文件/构建产物，确保后续轮次无状态污染）

**循环逻辑**：

```
1. 读取 task_plan.md，提取 Verification Commands 作为固定评估环境
2. 在 Task Space worktree 中，从工作分支创建 experiment/<slug>-<ts> 临时分支并切换
3. 运行基线评估 → 记录 baseline 到 .plans/<slug>/experiments.tsv（见 §3.4 存储归属）
4. FOR round = 1..N（每轮限时 5 分钟，见 §6.2）:
   a. Agent 基于实验历史 (experiments.tsv) 提出一个修改方案
   b. 委派执行（v1.0: 串行主上下文; Phase C1 后: 独立 Subagent 上下文）
   c. git commit -m "experiment-{round}: {description}"
   d. 运行评估 → 得到 new_pass_rate + optional metric_value
   e. 记录到 experiments.tsv（见 §3.4 TSV Schema）
   f. 决策（见 §3.5 评分合约）：
      - KEEP:  更新 baseline，继续下一轮
      - DISCARD: git reset --hard HEAD~1 && git clean -fd，继续下一轮
      - MARGINAL: 暂停，展示 diff + 指标 + 行数给用户决定
   g. 每轮结束展示本轮结果
5. 循环结束：切回工作分支，将所有 keep 的 commits cherry-pick 回来
6. 在工作分支上重新运行一次 Verification Commands（最终 Verify），确认净变更仍全绿
7. 删除实验临时分支
8. 写入实验摘要到 memory.jsonl（见 §3.6 实验记忆 Schema）
9. 如有净改进 → 提示 /alata:evolve extract（见 §3.7 与 evolve 的衔接）
10. 输出实验报告摘要
```

### 3.4 存储归属与 TSV Schema

**物理归属**：
- **统一状态根（Single Spine）**：所有共享状态都归属于 `state_root_path`，它同时拥有 `.alataflow/` 和 `.plans/`
- 默认情况下：`state_root_path == main_checkout_path`（主 checkout 是共享状态唯一真源）
- `experiments.tsv` 位于 `.plans/<slug>/experiments.tsv`（任务本地）
- `.plans/` 目录位于 `state_root_path/.plans/`，不跟随 worktree 分支
- `.alataflow/` 目录位于 `state_root_path/.alataflow/`，`spaces.json` / `current_space` / `session_state.json` / `memory.jsonl` / `error.log` / `experiment_active.*` 全部只存在这一份
- 实验循环期间，`experiment-tracker.js` 从 `state_root_path` 写入 TSV，**不在 worktree 分支上 commit 此文件**
- 并行 Task Space 各自写自己的 `.plans/<slug>/experiments.tsv`，无冲突
- `current_space` 在 v1.1 中降级为 **UX convenience pointer**：用于主 checkout 内的默认 Space 选择与状态展示；**不再**作为 hooks/runtime 判定“当前 worktree 属于哪个 Space”的权威来源

**前置条件——Space 元数据扩展**：
- `spaces.json` 中每个 Space 条目须包含 `state_root_path`、`main_checkout_path`（主 checkout 的绝对路径）和 `worktree_path`（worktree 的绝对路径）
- `space-manager.js` 的 `createSpace()` 在创建 worktree 时同时记录这三个路径
- 每个 worktree 根目录额外写入一个轻量指针文件 `.alataflow-root`，内容仅为 `state_root_path` 的绝对路径。它不是状态副本，只是路由指针
- 所有 hooks/runtime 先解析 `state_root_path`，再访问 `.alataflow/` 与 `.plans/`。解析顺序：
  1. 若当前目录存在 `.alataflow-root`，读取其内容
  2. 否则读取 `state_root_path/.alataflow/spaces.json` 或通过 `git worktree list --porcelain`，按 `cwd -> worktree_path -> slug` 反查
  3. 若 `cwd` 本身就是主 checkout 且同时存在 `.alataflow/` 与 `.plans/`，则取 `cwd`
  4. 若仍无法确定，**fail closed**：写 `error.log` 并跳过本次 hook/runtime 逻辑，绝不回退为在当前 worktree 新建一套 `.alataflow/`
- hooks/runtime 对“当前 Space”的解析顺序同样固定为：`cwd -> spaces.json[worktree_path] -> slug`；只有在主 checkout 且没有进入 worktree 时，才允许把 `current_space` 作为 UX fallback
- `experiment-tracker.js`、`progress-writer.js`、`memory-loader.js` 同理，始终通过 `state_root_path` 定位共享状态与计划文件
- `.alataflow-root` 仅作为未跟踪路由文件存在：加入 `.gitignore`（或 `.git/info/exclude`），不进入版本控制；删除 worktree 时同步删除
- **Phase A 前置任务**：在 A1 之前，先扩展 `space-manager.js` 的 Space 元数据 schema 与路径解析逻辑，补齐 `state_root_path` / `main_checkout_path` / `worktree_path` 字段（向后兼容：已有 Space 通过 `git worktree list --porcelain` 推导并回填）

**experiments.tsv Schema**：

```tsv
round	commit	description	pass_rate	metric_value	metric_direction	loc_delta	status	mode	timestamp
0	a1b2c3d	baseline	8/10	152.7	lower	0	baseline	repair	2026-03-22T10:00:00
1	d4e5f6g	优化数据库查询N+1	10/10	-	lower	-12	keep	repair	2026-03-22T10:05:00
2	h7i8j9k	添加缓存层	10/10	145.3	lower	+38	discard	normal	2026-03-22T10:12:00
3	l0m1n2o	简化中间件链	10/10	145.3	lower	-25	keep	normal	2026-03-22T10:18:00
4	p3q4r5s	过度优化尝试	10/10	142.1	lower	+80	marginal	normal	2026-03-22T10:25:00
```

| 列 | 说明 |
|---|---|
| `pass_rate` | 格式 `passed/total`，Layer 1 正确性门禁输入 |
| `metric_value` | `--metric` 命令的解析结果（浮点数）。`-` 表示：无 `--metric` 参数，或 Layer 1 已决策（pass_rate 改善/下降）而跳过 metric 评估 |
| `metric_direction` | `higher` 或 `lower`，无 metric 时为 `-` |
| `loc_delta` | 相对于 **最近一次 KEEP（或 baseline）** 的代码行数变化（`+N` 或 `-N`）。discard 的轮次不作为后续对比基准 |
| `status` | `baseline` / `keep` / `discard` / `marginal` / `timeout` / `error` |
| `mode` | `normal` (baseline 全绿) 或 `repair` (baseline 未全绿) |

### 3.5 评分合约（Score Contract）

决策基于 **两层判断**，严格按此顺序执行。**正确性永远优先于优化指标**。

**自定义 metric 规范**：
- 格式：`--metric "command" --metric-direction higher|lower`
- 解析：运行 command，取 stdout 最后一行，解析为单个浮点数（`parseFloat(lastLine)`）
- 解析失败（非数字输出、命令报错）→ 该轮标记 `error`，自动 DISCARD
- `--metric-direction` 必须显式指定，无默认值（防止方向歧义）

```
Layer 1: 正确性门禁 (Correctness Gate) — 二元通过/不通过
  输入: Verification Commands (task_plan.md 中定义，实验期间不可修改)
  指标: pass_rate = passed_count / total_count
  规则:
    - pass_rate < baseline_pass_rate → 自动 DISCARD，不进入 Layer 2
    - pass_rate > baseline_pass_rate → 自动 KEEP（正确性改善永远优先，即使 metric 下降）
    - pass_rate == baseline_pass_rate → 进入 Layer 2

  修复模式 (Repair Mode):
    当 baseline_pass_rate < 1.0 时自动激活。
    - pass_rate 改善（即使 < 1.0）→ KEEP（修复进展是最高优先级）
    - 日志标记 mode=repair，提醒用户基线未达全绿

Layer 2: 优化目标 (Optimization Objective) — 仅当 pass_rate == baseline 时进入
  输入: --metric 指定的命令输出（可选，默认无）
  规则:
    有 --metric 时:
    - metric 改善（按 --metric-direction）→ KEEP
    - metric 不变 + 代码行数减少 → KEEP（简洁性 tiebreaker）
    - metric 不变 + 代码行数不变/增加 → DISCARD
    - metric 微变（<5%，相对于最近 KEEP/baseline 的 metric_value）+ loc_delta 绝对值 > 当前代码总行数的 30% → MARGINAL（暂停，用户决定）

    无 --metric 时:
    - 代码行数减少 → KEEP（简洁性是默认优化目标）
    - 代码行数不变/增加 → DISCARD
```

**MARGINAL 处理**：实验循环暂停，向用户展示 diff、指标变化、行数变化，由用户选择 keep 或 discard。`/alata:experiment --rounds N` 是"可控自主"，不是无人值守。

**简洁性正则化**（Karpathy 精神）：Linus 三问自动嵌入每轮决策。

### 3.6 实验记忆 Schema（memory.jsonl 条目）

实验结束后，`experiment-tracker.js` 将关键发现写入 `memory.jsonl`，格式：

```json
{
  "kind": "experiment",
  "summary": "[EXPERIMENT] <slug>: <最佳 round 描述>, +<total_delta>, <kept>/<total> rounds kept",
  "content": "baseline pass_rate=8/10, best pass_rate=10/10, rounds=5, kept=2, mode=repair",
  "tags": { "project": "/path/to/canonical/project-root", "space": "<slug>", "type": "experiment" },
  "created_at": "ISO timestamp",
  "last_used_at": "ISO timestamp"
}
```

- `summary` 字段供 `memory-search.js` 全文匹配使用
- `tags.project` **固定写入 `state_root_path`**（主 checkout 的 canonical project root），绝不写 worktree cwd
- `memory-loader.js` / `session-start.js` 在 worktree 中运行时，先解析 `state_root_path`，再用该 canonical 路径作为 project filter，确保 recall 在主 checkout 与 worktree 下命中一致
- `/alata:recall` 搜索的是此摘要条目，不直接搜索 experiments.tsv 文件

### 3.7 与 evolve extract 的衔接

实验结束后 cherry-pick 回工作分支会产生新 commit。此时 `git diff HEAD` 可能为空（因为变更已 committed）。
因此实验循环在 cherry-pick 前记录 **pre-experiment baseline commit hash**，evolve extract 时使用：

```
git diff <baseline_commit>..HEAD  # 获取实验的净变更
```

- `experiment-tracker.js` 在 experiments.tsv round=0 行记录 baseline commit hash
- 最终 `/alata:verify` 在工作分支执行，验证的是 cherry-pick 后的净变更，而不是临时实验分支上的中间状态
- `/alata:evolve extract` 检测到 `experiments.tsv` 存在时，自动使用 baseline commit 做 diff
- Capsule 包含 experiments.tsv 摘要作为验证证据

### 3.8 实验模式下的 Hook 行为

现有 hooks 在自主多轮循环中会产生噪音或误导性副作用。实验循环需要显式适配：

| Hook | 正常模式 | 实验模式 (experiment_active=true) |
|------|---------|--------------------------------|
| `progress-save` (PostToolUse) | 每次 Write/Edit/Bash 追加 progress.md | **部分静默**：不写 progress.md，不增 write_count（实验进度由 experiments.tsv 记录）。但 **上下文水位检测仍然运行**——长实验循环正是最容易耗尽上下文的场景 |
| `stop-verify` (Stop) | 提示验证和 Capsule 提取 | **静默**：实验循环自行管理验证和结束流程 |
| `scrubber` (UserPromptSubmit) | 正常运行 | **正常**：安全扫描不跳过 |
| `session-start` (SessionStart) | 正常运行 | **正常**：不受影响 |

**实现方式**：
- 标记文件路径：`state_root_path/.alataflow/experiment_active.<slug>`（Space 级隔离，非全局单文件）
- 内容：JSON `{ "slug": "<slug>", "started_at": "ISO timestamp", "heartbeat_at": "ISO timestamp", "pid": <process.pid>, "timeout_sec": 300, "round": 2 }`
- 实验开始时，`experiment-tracker.js` 写入该标记文件
- `experiment-tracker.js` 在每轮开始、每轮评估结束时刷新 `heartbeat_at`；若单轮评估时间较长，则每 30 秒刷新一次 heartbeat
- `progress-save.js` 和 `stop-verify.js` 在解析出 `state_root_path` 后，必须通过 `cwd -> spaces.json[worktree_path] -> slug` 反查 **当前 Space**；仅在主 checkout 且不在任何 worktree 中时，才允许退回 `current_space`
- 若发现 **当前 Space 的** 标记文件存在，则跳过各自逻辑
- 实验结束时（正常结束或异常退出），删除标记文件
- **过期恢复**：`session-start.js` 启动时扫描 `state_root_path/.alataflow/experiment_active.*` 文件，按以下顺序判断是否为残留：
  1. 若 `pid` 仍存活（`process.kill(pid, 0)` 成功），保留标记，不做清理
  2. 若 `pid` 不存活，且 `heartbeat_at` 早于 `max(3 × timeout_sec, 900s)`，视为中断残留，删除标记并写入 `error.log`
  3. 若平台不支持 pid 存活检查，则退化为仅使用 `heartbeat_at` TTL 判断
- 标记文件在 `.gitignore` 中，不进入版本控制

## 五、双层优化模型（核心创新）

这是 Autoresearch 给 AlataFlow 最深层的启示。不只是加一个 `/experiment` 命令，而是重新定义整个系统的优化结构：

```
╔══════════════════════════════════════════════════════════════╗
║                    外层循环 (人类驱动)                         ║
║                                                              ║
║   人类优化：CLAUDE.md / Skills / 评估标准 / 委派模板           ║
║   频率：每周/每月                                             ║
║   依据：实验报告 + Capsule 成功率 + 团队反馈                   ║
║                                                              ║
║   ┌──────────────────────────────────────────────────────┐   ║
║   │                内层循环 (Agent 驱动)                   │   ║
║   │                                                      │   ║
║   │   Plan → Execute → Evaluate → Keep/Discard → Repeat  │   ║
║   │              ↑                        │               │   ║
║   │              └────── experiments.tsv ──┘               │   ║
║   │                                                      │   ║
║   │   频率：每个任务内多轮                                 │   ║
║   │   依据：Verification Commands (固定评估)               │   ║
║   └──────────────────────────────────────────────────────┘   ║
║                                                              ║
║   进化层 (跨任务学习)                                         ║
║   ┌──────────────────────────────────────────────────────┐   ║
║   │   Capsule: keep 的实验 → 提取 → 置信度追踪 → 复用      │   ║
║   │   Memory: 实验发现 → 索引 → 跨会话召回                  │   ║
║   │   Promotion: 高置信 Capsule → Skill 草稿              │   ║
║   └──────────────────────────────────────────────────────┘   ║
╚══════════════════════════════════════════════════════════════╝
```

**这意味着 AlataFlow 有三个时间尺度的优化**：

| 时间尺度 | 机制 | 优化什么 |
|----------|------|---------|
| **任务内** (分钟级) | `/alata:experiment` keep/discard 循环 | 当前任务的代码质量 |
| **跨任务** (天/周级) | Capsule 提取/复用 + 置信度衰减 | 组织的解决方案库 |
| **跨周期** (月级) | 人类优化 CLAUDE.md / Skills / 评估标准 | Agent 的工作策略 |

## 六、关键设计决策

### 6.1 不做什么（红线）

| 不做 | 理由 |
|------|------|
| TDD 铁律 | 非我所爱。Verification 已足够 |
| 6 平台支持 | 只做 Claude Code (主力) + Codex (可选) |
| 44+ 命令 | 保持 18 skills 封顶，克制是美德 |
| 无人值守全自动 | 软件开发需要人类判断。`/experiment --rounds N` 是可控自主 |
| SQLite / 外部数据库 | JSONL + TSV，零依赖，git-diffable |
| 多维加权评分 | 两层评分（正确性门禁 + 优化目标），决策输出三态 keep/discard/marginal。详见 §3.5 |

### 6.2 评估环境设计原则（Autoresearch 铁律）

直接照搬 Karpathy 的设计，结合 §3.5 评分合约：

1. **评估环境不可修改**：task_plan.md 中的 Verification Commands 一旦定义，实验循环中不可更改（防止 Goodhart's Law）
2. **两层评分，一个决策**：Layer 1 正确性门禁只比较 `pass_rate`；只有 `pass_rate == baseline` 时才进入 Layer 2 比较优化目标（`--metric` 或简洁性）。详见 §3.5
3. **固定时间预算**：每轮实验默认 5 分钟超时（`--timeout 300`，可配置）。超时 = 该轮自动 DISCARD + 记录 `status=timeout`
4. **简洁性是一等公民**：代码行数减少 + 指标不降 = 算改进（Layer 2 tiebreaker）
5. **崩溃是信息不是阻塞**：记录、诊断、继续
6. **可控自主，非无人值守**：`--rounds N` 在 KEEP/DISCARD 时自动推进，MARGINAL 时暂停等用户决定

### 6.3 与现有系统的关系

```
完整生命周期：Plan → Execute → [Experiment] → Verify → Review → Finish
                                    ↑ 可选插入点

/alata:experiment 在流程中的位置：
  - 位于 execute 之后、verify 之前
  - 不是 verify、review 或 finish 的替代品——验证、审查和合并仍然必经
  - 不是所有任务都需要 experiment——仅用于优化型任务

/alata:experiment 与 /alata:verify 的关系：
  - verify = 在工作分支运行一次最终验证，作为完成前的硬门
  - experiment = 在临时实验分支运行 N 轮「修改→评估→keep/discard」循环
  - experiment 内部会反复调用评估，但 cherry-pick 回工作分支后仍必须再跑一次 verify

/alata:experiment 与 /alata:evolve 的衔接：
  - experiment 结束 → 如果有净改进 → 提示 evolve extract
  - extract 使用 baseline commit diff（见 §3.7），不是 git diff HEAD

/alata:experiment 与 /alata:execute 的区别：
  - execute = 按计划执行步骤（线性推进）
  - experiment = 围绕一个目标反复迭代（RL 循环）
  - 典型用法：execute 完成后发现性能不达标 → experiment 迭代优化
```

## 十一、成功标准

v1.1 发布时应满足：

1. **闭环完整**：brainstorm → plan → execute → [experiment] → verify → review → finish（含 evolve extract）全链路跑通
2. **RL 循环可用**：`/alata:experiment --rounds 5` 能自主运行 5 轮迭代，产出 `.plans/<slug>/experiments.tsv`
3. **单一状态根成立**：无论在主 checkout 还是 worktree 中运行，`.alataflow/` 与 `.plans/` 都通过 `state_root_path` 统一解析，不出现双写或分叉
4. **实验 Git 隔离**：实验在一次性 `experiment/` 分支运行，keep 通过 cherry-pick 回工作分支，discard 通过 reset 丢弃，不留 revert 痕迹
5. **上下文不腐化**（Phase C 后）：execute/experiment 通过独立 Subagent 上下文委派，不污染主上下文
6. **实验可追溯**：每个实验有 commit hash + 评分 + keep/discard/marginal 状态
7. **知识不丢失**：实验关键发现以 `kind: experiment` 条目写入 memory.jsonl，下次 recall 可检索，且主 checkout/worktree 召回一致
8. **18 Skills / 12 Runtime / 5 Hooks**：不超标
9. **零外部依赖**：依然只需 Node.js，无 SQLite，无编译
