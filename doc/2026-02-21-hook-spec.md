# AlataFlow Hook 规格文档

**日期**：2026-02-21
**版本**：v1.0
**依赖**：Claude Code Hook 系统（15 事件，见官方文档）

---

## 设计原则

1. **薄 Hook**：Hook 只做事件路由，不写业务逻辑。业务逻辑全部在 Node.js runtime 脚本中。
2. **非阻塞优先**：除 scrubber（UserPromptSubmit）外，所有 Hook 失败一律 exit 0，不阻塞用户工作。
3. **防死循环**：Stop Hook 必须检查 `stop_hook_active` 字段，避免无限循环。
4. **最小读写**：每个 Hook 只读写自己职责范围内的文件，不越界。

---

## Hook 总览

| Hook | 事件 | 匹配条件 | 阻塞 | 职责 |
|------|------|---------|------|------|
| `session-start` | `SessionStart` | `startup, resume, clear, compact` | 否 | 记忆加载、环境自检、冷启动初始化、Capsule 衰减 |
| `scrubber` | `UserPromptSubmit` | 无（全量） | 是 | 隐私清洗，阻断含敏感内容的 prompt |
| `progress-save` | `PostToolUse` | `Write, Edit, Bash` | 否 | 自动持久化 progress.md，检查记忆阈值 |
| `stop-verify` | `Stop` | 无（全量） | 否 | 收尾提示（Capsule 提取、记忆同步） |

## Hook 1：`session-start`

### 触发条件
- 事件：`SessionStart`
- 匹配：`startup`、`resume`、`clear`、`compact`

### 职责（按顺序执行）

**Step 1 — 冷启动检测**
- 检查 `.alataflow/` 目录是否存在
- 不存在 → 调用 `runtime/init.js`，初始化目录结构：
  ```
  .alataflow/
  ├── memory.jsonl
  ├── spaces.json        （内容：[]）
  ├── session_state.json （内容：{}）
  └── error.log
  ```
- 输出欢迎信息至 `additionalContext`："[AlataFlow] 首次安装完成，已初始化工作目录。"

**Step 2 — 记忆加载**
- 读 `.alataflow/memory.jsonl`
- 按当前项目路径（`cwd`）过滤匹配条目（`tags.project` 字段）
- 取最近 10 条（按 `last_used_at` 降序）
- 汇总摘要注入 `additionalContext`

**Step 3 — Task Space 过期检查**
- 读 `.alataflow/spaces.json`
- 找出 `last_active` 超过 24h 且状态非 `completed` 的 Space
- 有则注入提醒至 `additionalContext`

**Step 4 — Capsule 月衰减（懒惰计算）**
- 读 `.alataflow/memory.jsonl` 中 `kind=capsule` 的条目
- 对 `last_used_at` 超过 30 天的条目：`confidence = confidence × 0.95`（下限 0.10）
- 更新写回 memory.jsonl（原地更新对应行）

### 读文件
- `.alataflow/memory.jsonl`
- `.alataflow/spaces.json`

### 写文件
- `.alataflow/session_state.json`（记录会话开始时间、cwd、project）
- `.alataflow/memory.jsonl`（仅 Capsule 衰减时更新）
- `.alataflow/`（仅冷启动时初始化）

### 输出格式
```json
{
  "additionalContext": "[AlataFlow] Loaded 3 memories for project 'my-ai-app'.\n- API 使用 FastAPI + Pydantic v2\n- 前端用 Next.js 14 App Router\n- 上次数据库迁移注意 UTF-8 编码\n⚠️ Space [auth-feature] 已闲置 26h，建议清理。"
}
```

### 失败处理
- 任何步骤抛出异常 → exit 0（非阻塞）
- 写错误详情至 `.alataflow/error.log`（格式：`ISO时间 [session-start] 错误信息`）
- 不向用户展示错误（避免干扰启动体验）

### 脚本路径
```
hooks/session-start.js
runtime/init.js
runtime/memory-loader.js
runtime/capsule-decay.js
```

## Hook 2：`scrubber`

### 触发条件
- 事件：`UserPromptSubmit`
- 匹配：无（全量，每次用户提交 prompt 均触发）

### 职责

**Step 1 — 敏感内容检测**

按优先级顺序运行检测规则：

| 规则 | 正则/方法 | 说明 |
|------|----------|------|
| API Key 模式 | `/(sk-[a-zA-Z0-9]{20,}|Bearer [a-zA-Z0-9+/=]{20,})/` | OpenAI / 通用 Bearer token |
| 高熵字符串 | Shannon entropy > 4.5，长度 > 20 | 可能是密钥或密码 |
| 私钥标记 | `/-----BEGIN (RSA\|EC\|OPENSSH) PRIVATE KEY-----/` | PEM 格式私钥 |
| 常见密码字段 | `/password\s*[:=]\s*\S{8,}/i` | 行内密码赋值 |

**Step 2 — 决策**
- 命中任一规则 → exit 2，stderr 输出提示（见下）
- 未命中 → exit 0，无输出

### 读文件
- 无（检测规则内置在脚本中）

### 写文件
- `.alataflow/scrubber.log`（命中时 append 一行，仅记录时间戳 + 命中规则类型，**不记录 prompt 内容本身**）

### 输出格式（阻断时）
```
exit 2
stderr:
[AlataFlow/Scrubber] ⚠️ 检测到可能的敏感内容（规则：高熵字符串）
请检查 prompt 是否包含 API Key、密码或私钥。
如确认内容安全，可在 prompt 前加 [SAFE] 标记跳过检测。
```

> **跳过机制**：用户在 prompt 开头加 `[SAFE]` 标记，scrubber 跳过检测直接放行。用于用户明确知道内容安全的场景。

### 失败处理
- 脚本自身 crash（语法错误、OOM 等）→ exit 0 放行
- 原则：**宁可放行，不可误杀正常工作**

### 脚本路径
```
hooks/scrubber.js
runtime/scrubber-rules.js
```

## Hook 3：`progress-save`

### 触发条件
- 事件：`PostToolUse`
- 匹配：`Write`、`Edit`、`Bash`

### 职责

**Step 1 — 读取当前 Task Space**
- 读 `.alataflow/current_space`（内容为 space slug，如 `auth-feature`）
- 无活跃 Space → 跳过，exit 0

**Step 2 — 写入 progress.md**
- 构造一行摘要（格式见下）
- Append 到 `.plans/<space-slug>/progress.md`
- **仅 Append，不覆盖**（并发安全的最小化保证）

摘要行格式：
```
[2026-02-21T10:30:00Z] [Write] src/auth/models.py (创建 User model)
```
其中描述部分从 `tool_input` 中提取（Write 取 `file_path`，Edit 取 `file_path`，Bash 取 `command` 前 80 字符）。

**Step 3 — 记忆阈值检查**
- 统计 `.alataflow/memory.jsonl` 行数
- 达到 50 条 → 在 `additionalContext` 中提示同步（不自动执行）
- 更新 `session_state.json` 中的 `write_count`（本轮写操作计数，供 Stop Hook 使用）

### 读文件
- `.alataflow/current_space`
- `.alataflow/memory.jsonl`（仅统计行数）
- `.alataflow/session_state.json`

### 写文件
- `.plans/<space-slug>/progress.md`（append）
- `.alataflow/session_state.json`（更新 `write_count`）

### 输出格式
```json
{
  "additionalContext": "[AlataFlow] Progress saved → .plans/auth-feature/progress.md"
}
```

记忆阈值提示时追加：
```json
{
  "additionalContext": "[AlataFlow] Progress saved.\n📊 记忆数：50/50，已达同步阈值。运行 /alata:memory sync 触发同步。"
}
```

### 失败处理
- 任何错误 → exit 0（非阻塞，工具已执行完毕）
- 写错误至 `.alataflow/error.log`
- 不影响用户工作流

### 脚本路径
```
hooks/progress-save.js
runtime/progress-writer.js
```

## Hook 4：`stop-verify`

### 触发条件
- 事件：`Stop`
- 匹配：无（全量）

### 职责

**Step 0 — 防死循环检查（必须第一步）**
- 读 `stop_hook_active` 字段
- 为 `true` → 立即 exit 0，不做任何操作

**Step 1 — 判断是否需要提示**

满足以下**所有条件**才提示（避免每次停止都打扰用户）：
- 有活跃 Task Space（`.alataflow/current_space` 非空）
- 本轮有写操作（`session_state.json` 中 `write_count > 0`）

**Step 2 — 构造提示**
- Capsule 提取提示（有写操作时）
- 记忆同步提示（memory.jsonl 行数 ≥ 45 条时，提前 5 条预警）

**Step 3 — 重置本轮计数**
- 将 `session_state.json` 中 `write_count` 归零

### 读文件
- `.alataflow/current_space`
- `.alataflow/session_state.json`
- `.alataflow/memory.jsonl`（仅统计行数）

### 写文件
- `.alataflow/session_state.json`（重置 `write_count`）

### 输出格式
```json
{
  "additionalContext": "[AlataFlow] 💾 本轮在 [auth-feature] 中有代码变更，可运行 /alata:evolve extract 提取 Capsule。"
}
```

记忆预警时追加：
```json
{
  "additionalContext": "...\n📊 记忆数：47/50，建议运行 /alata:memory sync 同步。"
}
```

### 失败处理
- 任何错误 → exit 0（绝不阻塞停止，防止 Stop Hook 死循环）
- 写错误至 `.alataflow/error.log`

### 脚本路径
```
hooks/stop-verify.js
runtime/stop-reporter.js
```

## 文件结构汇总

```
AlataFlow Plugin
├── hooks/
│   ├── hooks.json              # Hook 注册配置
│   ├── session-start.js        # Hook 1 入口
│   ├── scrubber.js             # Hook 2 入口
│   ├── progress-save.js        # Hook 3 入口
│   └── stop-verify.js          # Hook 4 入口
│
└── runtime/
    ├── init.js                 # 冷启动初始化
    ├── memory-loader.js        # 记忆加载与过滤
    ├── capsule-decay.js        # Capsule 月衰减
    ├── scrubber-rules.js       # 隐私清洗规则
    ├── progress-writer.js      # progress.md 写入
    └── stop-reporter.js        # 收尾提示生成
```

## hooks.json 注册配置（示意）

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": ["startup", "resume", "clear", "compact"],
        "hooks": [{ "type": "command", "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/session-start.js" }]
      }
    ],
    "UserPromptSubmit": [
      {
        "hooks": [{ "type": "command", "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/scrubber.js" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": ["Write", "Edit", "Bash"],
        "hooks": [{ "type": "command", "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/progress-save.js" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": "node ${CLAUDE_PLUGIN_ROOT}/hooks/stop-verify.js" }]
      }
    ]
  }
}
```

## 未使用的 Hook 事件（说明原因）

| 事件 | 不使用原因 |
|------|---------|
| `PreToolUse` | 权限控制交给 Claude Code 原生机制，AlataFlow 不做二次拦截 |
| `PermissionRequest` | 同上 |
| `PostToolUseFailure` | v1.0 不做失败自动恢复，由用户决策 |
| `SubagentStart` / `SubagentStop` | v1.1 并发探索时再引入 |
| `PreCompact` | v1.1 strategic-compact skill 实现时再引入 |
| `SessionEnd` | 收尾逻辑在 Stop Hook 处理，无需重复 |
| `Notification` / `ConfigChange` / `TeammateIdle` / `TaskCompleted` | v1.0 不涉及 |
